# Ch2 "수조의 탑" 설계서

## 요약

원래 스펙의 Ch2 "저울의 탑"(양팔저울 퍼즐)을 **수조(水槽) + 수위 센서**
메커닉으로 교체한다. 저울은 "양쪽을 같게 맞추기"라는 퍼즐 구조 특성상
**플레이어가 한쪽을 보고 똑같이 배치하는 단조로운 풀이**에 빠지기 쉽고,
얼리기가 필수가 되도록 강제하기 어려웠다.

수조 방식은 **절대적 목표(빨간 선에 닿기)**를 제시하고, **얼리기로 부피를
팽창시키는 것이 유일한 해법**이 되는 퍼즐을 자연스럽게 설계할 수 있다.
동시에 **수조 옆 전광판의 "무게" 수치가 불변**으로 유지됨을 시각적으로
강조해 "부피↑ / 무게=" 핵심 개념을 aha 모먼트로 전달한다.

## 교육 목표

- **핵심**: 물이 얼면 부피가 커지지만, 무게는 변하지 않는다 (부피 ↑ / 무게 =)
- **보조 1**: 상태 변화는 가역적이다 (얼리기 ↔ 녹이기)
- **보조 2**: 무게는 물질의 양(drops 수)에만 의존한다 — 얼려도, 녹여도, 모양이 바뀌어도 같다

## 1. 신규 메커닉

### 1.1 수조 (Tank)

**격자 타일 타입 추가**: `ground.type = 'tank'`

- 격자에선 **단일 셀**을 차지한다. 하지만 렌더링 시 **세로 2~3 타일 높이**로 확대 그려진다 (시각적으로 큰 잔 형태).
- **주인공 통행 불가**: 수조 타일은 `canEnter` 규칙에서 벽·물샘·모닥불과 같이 거부된다 (주인공은 수조 위에 올라갈 수 없다).
- 각 수조는 독립적인 **내부 상태**를 가진다:
  ```typescript
  interface TankState {
    position: Position;
    contentType: 'empty' | 'water' | 'ice';
    drops: number;          // 지금까지 부어진 물 방울 수
    threshold: number;      // 센서 임계치 (레벨 YAML에서 지정)
  }
  ```
- 부피 계산 (파생값):
  - `contentType === 'empty'`: 부피 = 0
  - `contentType === 'water'`: 부피 = drops × 10
  - `contentType === 'ice'`: 부피 = drops × 12 *(20 % 팽창)*
- 무게 계산 (파생값):
  - 무게 = drops *(상태 무관, 항상 drops 수와 같음)*

### 1.2 수위 센서 (Water Level Sensor)

- 각 수조의 `threshold` 값이 빨간 선 높이를 결정한다.
- **센서 활성 조건**: `current volume >= threshold`
- 수조 그래픽에 **빨간 수평 선**으로 임계치 표시.
- 부피가 선 위로 올라가면 색상 하이라이트 (시각 피드백).

### 1.3 게이트 (Gate)

**새로운 오브젝트**: `object.type = 'gate'`

```typescript
interface GateObject {
  type: 'gate';
  id: string;             // 레벨 YAML에서 지정
  linkedTankIds: string[];  // 하나 이상의 수조 id. 모두 활성돼야 열림 (AND)
}
```

- 닫힘 상태: 주인공 통과 불가, 얼음 팽창 통과 불가
- 열림 상태: 플로어처럼 취급 (주인공 통과 가능, 얼음 팽창 통과 가능)
- 연결된 수조(들)의 센서가 **전부 활성**이어야 열린다.
- 센서가 다시 비활성이 되면 문이 **다시 닫힌다** (Ch2-2 "녹으면 닫혀" 학습의 근거).

### 1.4 전광판 (Readout)

- 수조 바로 아래에 **작은 숫자 2행** 표시:
  ```
  부피: 10
  무게: 1
  ```
- 상태 변화 시 **짧은 tween 애니**로 변화 강조 (예: water 1방울 → ice 1덩이 시 부피 10 → 12로 숫자 점프, 무게는 그대로 1).
- 무게 숫자는 **절대 바뀌지 않는 것을 눈으로 확인**하도록 볼드·고정 위치·큰 폰트.

## 2. 상호작용 모델 (기존 모드 재활용)

기존 2-모드(`play` / `melt`)를 그대로 쓰되, 수조 탭에 대한 분기만 추가.

### 2.1 👣🧊 이동·얼리기 모드에서 수조 탭

조건별 처리 (탭한 수조와 주인공 인접 필요):

| 수조 상태 | 주인공이 물샘 인접? | 동작 |
|----------|-------------------|------|
| `empty` | O | **물 1방울 주입** (contentType → 'water', drops = 1) |
| `empty` | X | 무효 (아무 변화 없음) |
| `water` | (무관) | **얼리기** (contentType → 'ice', 부피만 20 % 팽창) |
| `ice` | (무관) | 무효 (녹이기는 다른 모드) |

물샘 인접 + 물 있는 수조 탭이 겹치는 경우(=물 더 붓기)는 **얼리기가 우선**한다. "물 더 붓기"는 비어있는 수조에 한정해 단순화한다 (필요시 Ch3+에서 "재충전" 메커닉 도입).

### 2.2 🔥 녹이기 모드에서 수조 탭

| 수조 상태 | 동작 |
|----------|------|
| `ice` | **녹이기** (contentType → 'water', 부피 -20 %) |
| `water` | **비우기 (drain)** — contentType → 'empty', drops = 0 (세부 정당성: 3.1절) |
| `empty` | 무효 |

### 2.3 환경 효과 (모닥불 자동 녹임)

- 모닥불 4방향 인접 수조는 **매 턴 자동 녹음**:
  - `ice` 상태면 → `water`로 (drops 유지, 부피 수축)
  - `water` 또는 `empty`면 변화 없음
- 주인공이 **수조 바로 옆 칸**에 서 있어도 자동 녹임은 **적용됨** (수조는 주인공이 "밟고 있는" 타일이 아니기 때문). Ch1의 "주인공 발밑 얼음은 녹지 않음" 규칙은 수조엔 해당 없음.

### 2.4 턴 소비

- 물 붓기 / 얼리기 / 녹이기 각각 **1 턴 소비**
- 실패(무효 행동)는 턴 소비 없음 (기존 규칙과 일치)

## 3. 스테이지 구성 (8 스테이지)

### 공통 설계 원칙

- 각 스테이지는 **한 가지 새 메커닉 포인트**를 도입하고, 이전에 배운 걸 반복 활용.
- 턴 수 제한 없음. Undo 무제한. 학생 실수 부담 ↓.
- 스테이지 클리어 조건: 모든 게이트 통과 → 출구 E 도달. 꽃은 Ch2에선 거의 사용하지 않음 (수조 퍼즐에 집중).
- 각 스테이지 크기: 7×5 ~ 12×8 격자.

### 2-1 "첫 수조"
- **학습**: 수조에 물 붓기 → 얼리기 → 부피 팽창으로 센서 도달
- **레이아웃**: 수조 1개, 센서 임계치 = 12 (물 1방울=10은 부족, 얼음 1방울=12는 충분)
- **해법 순서**: 물샘 옆 → 수조에 물 1방울 → 수조 얼리기 → 게이트 열림 → 출구
- **예상 턴**: 4~5

### 2-2 "녹으면 닫혀"
- **학습**: 가역성. 얼음이 녹으면 부피 줄어 센서가 다시 꺼지고 문 닫힘
- **레이아웃**: 2-1과 유사하나 게이트 너머로 가기 전에 **경로상 모닥불 근처**를 통과해야 함 — 근데 얼린 수조가 모닥불 옆에 있으면 녹음
- **해법**: 수조를 얼려 문 연 뒤 **빠르게 통과** (모닥불이 수조를 녹이기 전에)
- **예상 턴**: 6~8

### 2-3 "무게가 모자라"
- **학습**: 부피 팽창만으론 부족. 더 큰 부피가 필요하면 **drops를 더 부어야 함** = 무게(drops)는 단순 증가
- **레이아웃**: 수조 1개, 센서 임계치 = 24 (물 2방울=20 부족, 얼음 2방울=24 충분)
- **제약**: 물을 한 번 부은 뒤 "비어있지 않은" 수조에는 더 못 부음. 그러니 **순서 설계 필요**:
  - 물 1방울 → 얼리기 → 녹이기(다시 비움)? NO, 녹이면 물로 돌아가 비어있지 않음.
  - **실제 해법**: 2-3에서는 수조가 2개일 수 있다 — 한 수조당 1방울만 받고, 얼려 연결. 또는 수조 메커닉 확장: "녹인 물은 드레인해서 비울 수 있다" 같은 새 기믹 — 설계 확정 필요.
- **대안 설계**: **"비우기(drain)" 버튼**을 수조에 추가. 물 상태 수조를 비우면 drops → 0으로 리셋. 그러면 2방울이 필요한 센서는 한 수조에 연속으로 못 붓는 구조가 되고, **수조 2개 협업** 또는 **여러 수조에 분배** 전략이 필요해진다.
- *(참고: 세부 메커닉 확장은 이 스펙의 3.1절에서 별도 정리)*

### 2-4 "움직이는 센서"
- **학습**: 타이밍. 센서가 몇 턴 간격으로 위아래 움직임. 얼린 수조 부피와 센서 높이가 맞아야 게이트 열림
- **레이아웃**: 수조 1개 + 주기적으로 움직이는 센서 (매 N턴마다 임계치 ±값)
- **해법**: 특정 턴에 얼린 상태여야 열림. 타이밍 맞춰 얼리기 / 녹이기
- **예상 턴**: 8~12

### 2-5 "두 수조"
- **학습**: 자원 분배 (수조 2개 독립 관리)
- **레이아웃**: 수조 2개, 게이트 2개 (각기 다른 수조에 연결). 둘 다 열어야 출구 접근
- **해법**: 물샘에서 각 수조에 물 붓고, 각기 다른 타이밍에 얼리기
- **예상 턴**: 10~14

### 2-6 "모닥불 광장"
- **학습**: 환경 변수. 모닥불 배치로 수조가 자동 녹음
- **레이아웃**: 수조 2개 중 하나가 모닥불 인접. 그 수조는 얼려도 다음 턴이면 녹음
- **해법**: 모닥불 옆 수조가 얼린 상태를 유지하려면 다른 방법 (모닥불 자체 끄는 기믹? 아님 적극 녹이기 타이밍?) — **최종 설계: 모닥불 옆 수조는 1턴만 얼음으로 있다가 녹음. 그 "1턴 짜리 얼음 상태" 동안 게이트 열고 통과해야 함**
- **예상 턴**: 10~15

### 2-7 "연결된 수조" *(선택, 구현 복잡도 높음)*
- **학습**: 심화 — 한 수조의 변화가 다른 수조에 영향
- **레이아웃**: 2개의 수조가 "관(pipe)" 타일로 연결됨. 한 쪽에 물 붓거나 녹이면 수위가 평형을 맞추려 다른 쪽으로 흘러감
- **구현 복잡도 크므로 Phase 최종 단계에서 결정**. Phase 3 초안에서 스킵 가능.

### 2-8 "탑 꼭대기" (피날레)
- **학습**: 종합
- **레이아웃**: 수조 3개 + 모닥불 1~2개 + 게이트 2~3개 + 움직이는 센서 1개
- **예상 턴**: 15~20, 4학년 최상위 학생 도전 수준

## 3.1 세부 메커닉 확장: 수조 "비우기(drain)"

2-3 "무게가 모자라" 퍼즐을 위해 추가 기능:

**새 액션**: `drain` — 수조의 내용을 모두 비워 `contentType='empty', drops=0`으로 리셋.

**트리거**: 수조 탭 시 현재 모드와 별개로, **수조가 '물' 상태일 때 녹이기 모드 탭 시 비우기**로 동작 (녹이기 모드 × 물 = 비우기 — 얼음 상태에선 녹이기).

수정된 상호작용 매트릭스:

| 모드 | 수조 상태 | 동작 |
|------|----------|------|
| 이동·얼리기 | empty + 물샘 인접 | 물 1방울 주입 |
| 이동·얼리기 | water | 얼리기 |
| 이동·얼리기 | ice | 무효 |
| 녹이기 | empty | 무효 |
| 녹이기 | water | **비우기** (drain → empty) |
| 녹이기 | ice | 녹이기 (ice → water) |

학습 의미: "녹이기 모드에서 물을 탭하면 물을 쏟아버린다" → 4학년 이해 자연스러움 ("녹여서 흘려보냄").

## 4. 기술 아키텍처 통합

### 4.1 추가되는 타입

```typescript
// src/core/types.ts 확장
export type GroundType =
  | 'floor' | 'wall' | 'spring' | 'bonfire' | 'exit'
  | 'tank';          // 신규

export interface TankState {
  position: Position;
  contentType: 'empty' | 'water' | 'ice';
  drops: number;
  threshold: number;
  id: string;
}

export interface GateObject {
  type: 'gate';
  id: string;
  linkedTankIds: string[];
  open: boolean;
}

export type ObjectCell =
  | WaterObject | IceObject | BoxObject | RockObject | FlowerObject
  | GateObject;     // 신규

export type ActionKind =
  | { kind: 'move'; direction: Direction }
  | { kind: 'pour'; target: Position }
  | { kind: 'freeze'; target: Position; direction: Direction }
  | { kind: 'melt'; target: Position }
  | { kind: 'pourTank'; target: Position }     // 신규
  | { kind: 'freezeTank'; target: Position }   // 신규
  | { kind: 'meltTank'; target: Position }     // 신규
  | { kind: 'drainTank'; target: Position };   // 신규
```

### 4.6 솔버 (선택)

**목표**: Ch2 레벨 YAML 작성 후 **자동으로 풀이 가능성 검증** — BFS로 상태 공간 탐색.

- 상태 = `{ player position, tank states, gate open flags }`
- 액션 = 위 ActionKind 전부
- 승리 = player position === exit position AND flowers collected OK
- 구현 우선순위: 낮음 (Phase 3 후반). 레벨 디자인 시 손으로 풀 수 있으면 우선 배포.

## 5. UI/UX 조정

### 5.1 수조 렌더링

- 격자 타일 위에 **세로 2칸 높이 수조 스프라이트** 그림 (렌더링만; 격자 상으론 1칸)
- 내부에 **수위 바 (파란/하늘색 채움)**, 상단에 **빨간 선 (threshold 마커)**
- 아래에 **전광판 텍스트 2줄**:
  ```
  부피: 12
  무게: 1
  ```
- 상태 변화 시 **tween 애니** (bar 높이 tween, 숫자 카운터 tween)

### 5.2 게이트 렌더링

- 닫힘: 나무문/얼음빗장 그래픽, 어두운 색
- 열림: 반투명·밝은 색, 통과 가능 표시
- 게이트 바로 옆에 **연결된 수조 id** 소형 표시 (예: "→t1") — 디버깅 및 학생 힌트용

### 5.3 탭 힌트 (선택)

스테이지 진입 시 첫 수조 위에 짧게 깜빡이는 "탭 해보세요 👆" 같은 튜토리얼 (2-1에만).

### 5.4 전광판 색상 강조

- 무게 숫자는 **항상 같은 색** (예: 흰색). 변하지 않는 것을 체감
- 부피 숫자는 상태에 따라 색 변화 (예: 물=파랑, 얼음=하늘색). 시각적으로 "변하는 값"과 "불변 값" 구분
