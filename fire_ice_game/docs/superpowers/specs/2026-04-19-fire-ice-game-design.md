# 얼음공주 × 불왕자 — 게임 설계 문서

**프로젝트명:** Fire & Ice Quest (얼음공주 × 불왕자)
**작성일:** 2026-04-19
**대상 학년/단원:** 초등 4학년 1학기 과학 2단원 "물의 상태변화"
**형식:** 2인 협동 퍼즐 플랫포머 (Fireboy & Watergirl 오마주)
**스코프:** MVP — 4스테이지 · 총 플레이 35~45분

---

## 1. 프로젝트 개요

초등 4학년 과학 2단원 "물의 상태변화"의 핵심 개념(녹음/얾/끓음/증발/응결/부피팽창)을 **직접 게임 메카닉**으로 체험하는 2인 협동 퍼즐 게임이다. 학생 두 명이 각자 태블릿을 들고 **Wi-Fi로 실시간 연결**되어, 얼음공주(얼리기/응결)와 불왕자(녹이기/끓이기) 능력을 **합쳐야만** 퍼즐을 풀 수 있다.

상태변화 개념이 각 스테이지의 핵심 메카닉으로 작동하므로, 학생은 "교과서로 배우는 게 아니라 게임으로 경험"하게 된다.

### 이 프로젝트가 해결하려는 것

- 4학년 과학 상태변화 단원은 실험 기반이지만 교실에서 모든 현상(특히 부피 팽창, 응결)을 바로 관찰하기 어려움
- 추상적 개념을 **시각적·조작적**으로 체감하게 해주는 보조 자료가 부족
- 협동 학습 장면이 필요한 시간에 "교과 내용과 연관된" 활동이 드묾

### 성공의 정의

- 학생 2인이 Wi-Fi로 연결해 4스테이지를 끝까지 함께 플레이할 수 있음
- 각 스테이지가 교과서 학습 목표 1~2개와 직접 맞닿아 있어, 클리어 후 "왜 그게 통했는지" 학생이 설명할 수 있음
- 수업 1차시 또는 자유활동 시간 한 세션 안에 끝남 (35~45분)

### 비목표 (MVP 범위 밖)

- 3명 이상 멀티플레이
- 온라인/원격 매칭 (같은 학교 Wi-Fi 내부만)
- 계정·로그인·진도 저장 (세션 종료 시 휘발)
- 사운드/BGM 완성도 (효과음 최소한만)
- 스테이지 에디터
- 리더보드/랭킹
- 5번째 이상 스테이지, 보스 로스터 확장
- 모바일 앱 스토어 배포 (브라우저로 접속)

## 2. 타깃 사용자 및 환경

### 사용자

- **플레이어**: 초등 4학년 학생 2인
- **운영자**: 담임 교사 (서버 실행, 태블릿 분배, 진행 안내)

### 하드웨어 / 네트워크

- 태블릿 2대 (iPad 또는 Android 태블릿)
- 서버: 교사 노트북 (Windows/Mac, Node.js 설치)
- 네트워크: 교사 노트북 또는 스마트폰을 **AP(핫스팟)로 활용** — 학교 Wi-Fi의 Client Isolation 우회
- 접속: 태블릿 브라우저에서 `http://<서버IP>:<포트>` 직접 입력

### 운영 시나리오

1. 교사가 노트북에서 `npm start` 실행 → Express 서버 기동
2. 교사가 핫스팟 켜고 태블릿 2대 접속
3. 태블릿 둘 다 브라우저에서 서버 주소 입력 → 자동으로 Player 1 / Player 2 역할 배정
4. 캐릭터 선택 화면에서 각자 얼음공주/불왕자 선택
5. Stage 1부터 순차 진행

## 3. 기술 스택

| 레이어 | 선택 | 이유 |
|---|---|---|
| 서버 | Node.js + **Express** | 정적 파일 서빙, 표준 |
| 실시간 통신 | **Socket.IO** | 방(room) 관리, 재연결, 브로드캐스트 내장 |
| 클라이언트 렌더링 | **HTML5 Canvas** + React (UI) | 인트로 프로토타입과 일관성. Canvas로 게임 루프, React로 HUD/메뉴 |
| 물리/충돌 | 자체 구현 (AABB + 타일 그리드) | 스코프 작아 경량 엔진 불필요 |
| 스프라이트 | SVG 픽셀 그리드 (기존 인트로에서 재활용) | 픽셀아트 스타일, 해상도 독립 |
| 에셋 포맷 | 스테이지는 JSON (타일 맵) | 읽기 쉬움, 확장 용이 |

### 왜 이 조합인가

- Socket.IO + Express는 한글 자료 풍부, 교사 본인도 유지보수 가능한 수준
- Canvas가 태블릿 터치에서 60fps 안정적
- 기존 인트로(`index.html`)가 이미 React + SVG 픽셀 그리드라 **자연스럽게 연결**
- 타이틀/메뉴/캐릭터 선택/스토리 화면은 **기존 인트로 그대로 재사용**, 게임 씬(GameScene)만 추가

### 대안 평가

- **Phaser/Godot**: 오버킬. 1~2개 씬 짜는 데 프레임워크 학습비 큼
- **WebRTC P2P**: 신호 서버 여전히 필요, NAT 트래버설 골치. 교실 환경에서 서버 있는 편이 차라리 단순
- **Colyseus**: 게임 전용 서버 프레임워크. 좋지만 한글 자료 적어 나중 확장 시 고려

## 4. 아키텍처 개요

### 클라이언트 ↔ 서버 관계

```
[Tablet 1]  ──(socket)──>  [Server: Node.js]  <──(socket)──  [Tablet 2]
   ↑                            │                                  ↑
   └── 입력 전송                 │ 게임 상태 계산 (authoritative)     └── 입력 전송
       (조이스틱/버튼)            │ 30fps로 상태 브로드캐스트
   ← 렌더링 상태 수신 ←──────────┴────────────→ 렌더링 상태 수신 →
```

- **서버가 권한 있음 (authoritative)**: 물리·충돌·게임 로직은 서버에서 단일하게 계산. 치팅 방지 + 두 태블릿 간 불일치 제거
- 클라이언트는 **입력만 전송**하고 **받은 상태를 렌더링**
- 클라이언트 사이드 예측(client-side prediction) 간단히 적용: 본인 캐릭터 이동은 즉시 반영, 서버 응답 오면 보정

### 데이터 플로우

**입력 (client → server, 약 30Hz)**
```json
{ "type": "input", "keys": { "left": true, "right": false, "jump": false, "ability": false } }
```

**상태 (server → clients, 30Hz 브로드캐스트)**
```json
{
  "type": "state",
  "tick": 1234,
  "players": [
    { "id": "P1", "role": "ice", "x": 320, "y": 480, "vx": 0, "vy": -2, "hp": 3, "facing": "right" },
    { "id": "P2", "role": "fire", "x": 360, "y": 480, "vx": 1, "vy": 0, "hp": 3, "facing": "left" }
  ],
  "entities": [
    { "id": "ice_block_1", "type": "ice", "x": 400, "y": 500, "state": "solid" },
    { "id": "steam_col_1", "type": "steam", "x": 500, "y": 400, "height": 200, "ttl": 5000 }
  ],
  "stageStatus": { "stageId": 1, "beat": 2, "completed": false }
}
```

### 방(room) 관리

- 서버 기동 시 방 1개 자동 생성 (MVP에서는 단일 방)
- 첫 접속자 = P1, 두 번째 접속자 = P2
- P1 disconnect 시 게임 일시정지, 재접속 대기
- 모든 플레이어 이탈 → 5분 후 방 리셋

## 5. 게임 디자인

### 5.1 캐릭터 능력

| 캐릭터 | 담당 변환 | 버튼 효과 | 패시브 |
|---|---|---|---|
| 🔵 **얼음공주 엘사벨** | 물→얼음 (얾) / 수증기→물 (응결) | 전방에 냉기 분출, 물이면 얼음으로, 수증기면 물로 | 얼음 위 이동속도 +20%, 수증기 노출 시 HP 감소 |
| 🔴 **불왕자 플레이든** | 얼음→물 (녹음) / 물→수증기 (끓음) | 전방에 불꽃 발사, 얼음이면 물로, 물이면 수증기로 | 불꽃 경로 위 이동속도 +20%, 물 접촉 시 HP 감소 |

양방향 변환이 각자 2개씩, 겹치지 않으므로 **협동이 강제**된다.

### 5.2 공통 조작 (태블릿)

```
┌─────────────────────────────────────┐
│  [🎮 좌측 가상 조이스틱]    [⬆ 점프] │  세로로 배치
│                            [⚡ 능력] │
└─────────────────────────────────────┘
```

- 좌측 하단: 가상 조이스틱 (이동)
- 우측 하단: 점프 버튼, 능력 버튼 (세로 2개)
- 상단 우측: HP 표시, 일시정지 버튼
- 상단 좌측: 현재 스테이지/비트 표시

### 5.3 공통 규칙

- HP: 각 3. 0 도달 시 3초 뒤 체크포인트에서 부활
- 체크포인트: 비트 시작 시점마다 자동 저장
- 타일 크기: 32×32px
- 중력: 600 px/s², 점프 초기속도 -340
- 이동 속도: 기본 180 px/s

### 5.4 스테이지 설계 (4개)

#### Stage 1 — 녹는 얼음 다리 (녹음 · 얾)

- **오프닝**: 얼음 왕국 남문, 얼어붙은 "여명의 개울". NPC 눈꽃 요정 플루가 튜토리얼
- **Beat 1**: 얼음 기둥을 왕자가 녹여 물로, 공주가 얼려 다리 만들어 건너기
- **Beat 2**: 고드름 녹이면 물 낙하 → 공중에서 공주가 얼려 발판 (타이밍 2초 여유)
- **Beat 3**: 두 메카닉 조합 응용, 출구 도달
- **클리어 조건**: 두 캐릭터 모두 출구 도달
- **플레이 시간**: 5~8분
- **학습 포인트**: 녹음·얾은 반대 방향의 가역 변화

#### Stage 2 — 수증기 풍차 (끓음 · 증발)

- **오프닝**: 초원, 멈춘 풍차. NPC 장인 덩구
- **Beat 1**: 왕자가 물 끓여 수증기 생성 → 풍차 1대 돌려 문 열기
- **Beat 2**: 수증기 엘리베이터로 수직 상승, 공주는 수증기 3초 노출 HP 감소
- **Beat 3**: 풍차 3대 동시 가동 협동 퍼즐 (왕자=끓이기, 공주=물공급+벽설치)
- **클리어 조건**: 3풍차 모두 회전 + 문 통과
- **플레이 시간**: 7~10분
- **학습 포인트**: 끓음·증발의 차이, 수증기 부력

#### Stage 3 — 얼리면 깨진다 (얾 · 부피 팽창)

- **오프닝**: 바위산 협곡, 고대 석조 봉인문. 그림자 탑이 한쪽 하늘 가득
- **Beat 1**: 바위 크랙에 물 채우고 얼려 폭파 (핵심 메카닉 소개)
- **Beat 2**: 수직 크랙 — 공주가 댐 설치 → 왕자가 물 공급 → 공주가 얼림 (순서 퍼즐)
- **Beat 3**: 크랙 기둥 3쌍 동시 폭파, 30초 타이머 + 2초 동시 허용 창
- **클리어 조건**: 봉인문 개방
- **플레이 시간**: 10~12분 (난이도 피크)
- **학습 포인트**: 물이 얼면 부피 약 9% 증가 (교과서 실험)

#### Stage 4 — 물방울의 귀환 (응결 · 종합)

- **오프닝**: 그림자 탑 내부, 탑의 심장이 박동
- **Beat 1**: 수증기 방 탈출 — 공주가 차가운 패널로 응결시켜 수증기 제거
- **Beat 2**: 순환 미로 — 얼음→물→수증기→응결→얼음 체인을 한 방에서 경험
- **Beat 3**: 보스 "그림자 군주 녹타" 전투 — 패턴 3종 카운터
- **Beat 4**: 탑의 심장 정화 — 4단계 상태변화 순차 주입 최종 퍼즐
- **클리어 조건**: 탑의 심장 정화
- **플레이 시간**: 12~15분
- **학습 포인트**: 응결, 물의 순환 통합

### 5.5 네트워크 친화적 퍼즐 규칙

모든 "동시" 혹은 "타이밍" 퍼즐은 최소 **2초 허용 창**을 가지며, 실패해도 **리트라이 가능** (즉사 없음). 네트워크 지연 100~200ms 환경에서도 공정하게 플레이 가능해야 함.

## 6. 아트 / UX

### 아트 방향

- **픽셀아트** — 기존 인트로 A 방향(CRT 레트로) 기조 유지
- 팔레트: 얼음(민트·블루) / 불(코랄·오렌지) 대비 + 탑(보라 어둠)
- 폰트: Galmuri11 (픽셀 한글)
- 캐릭터: 기존 엘사벨/플레이든 SVG 스프라이트 그대로 활용 (인트로 `components/sprites.jsx`)

### 연출

- 상태변화 발생 시 **작은 파티클 + 아이콘 팝업** (녹음 🔥→💧, 얾 ❄→🧊, 끓음 💧→💨, 응결 💨→💧)
- 스테이지 클리어 시 **상태변화 도감 페이지 1장 획득** 애니메이션
- 배경에 **그림자 탑**이 스테이지마다 더 크게 보이게 (Stage 1 = 실루엣, Stage 4 = 정면)

### HUD (게임 중)

```
┌────────────────────────────────────────────────┐
│ Stage 1 · Beat 2               P1 ♥♥♥  P2 ♥♥♥ │
│ 여명의 개울                                 ⏸  │
│                                                 │
│  [게임 영역]                                    │
│                                                 │
│                                                 │
│  [🎮]                                [⬆]       │
│  ← 이동 →                             [⚡] 능력 │
└────────────────────────────────────────────────┘
```

## 8. 컴포넌트 단위와 책임

각 유닛은 하나의 명확한 책임만 갖고, 다른 유닛과는 잘 정의된 인터페이스로 통신한다.

### 서버

- **`server.js`** — HTTP/소켓 초기화, 정적 파일 서빙. 게임 로직 없음
- **`room.js`** — 플레이어 접속·이탈 처리, 방 수명주기. `gameState`의 입력·출력만 알고 그 내부는 모름
- **`gameState.js`** — 순수 리듀서: `state, inputs → nextState`. 네트워크·I/O 몰라야 함
- **`physics.js`** — AABB 충돌 + 타일 그리드 이동. `gameState`가 호출
- **`stages/*.json`** — 타일 맵 + 비트 트리거 정의. 로직 없음, 데이터만

### 클라이언트

- **`net.js`** — Socket.IO 연결·재연결. 이벤트를 게임 상태로 노출
- **`game.js`** — 렌더링 루프. `net.js`에서 받은 상태를 Canvas에 그림
- **`entities/`** — 각 엔티티 타입별 렌더러. 상태만 받아 그림 (상태 변경 안 함)
- **`HUD.jsx`** — React 컴포넌트, HP/스테이지 정보 표시

### 테스트 가능성

- `gameState.js`와 `physics.js`는 순수 함수라 단위 테스트 쉬움
- `room.js`는 Socket.IO 모킹하여 통합 테스트
- 전체 게임플레이는 수동 테스트 (태블릿 2대 시나리오)

## 9. 단계적 구현 제안

이 문서는 **설계**이며, 구현 계획은 별도 `plan` 문서에서 상세화. 큰 그림은:

1. **Phase 1** — 서버/클라 골격 + 빈 GameScene + 인트로 연결
2. **Phase 2** — Stage 1 (녹음/얾 메카닉 검증)
3. **Phase 3** — Stage 2, 3, 4 순차 확장
4. **Phase 4** — 교실 현장 테스트 + 버그 수정

각 Phase는 독립적으로 플레이 가능해야 함 (Phase 2 끝나면 Stage 1만이라도 둘이서 할 수 있음).

## 10. 리스크 및 오픈 이슈

| 리스크 | 완화 방안 |
|---|---|
| 학교 Wi-Fi Client Isolation | 교사 스마트폰/노트북 핫스팟 사용 |
| 네트워크 지연으로 타이밍 퍼즐 실패 | 모든 타이밍 창 ≥2초, 즉사 없음 |
| 태블릿 브라우저 성능 차이 (구형 아이패드) | Canvas 해상도 자동 조정, 60→30fps fallback |
| 두 플레이어 중 한 명 이탈 | 일시정지 + 재접속 대기, 5분 타임아웃 시 방 리셋 |
| 학생이 서버 IP 입력 어려워함 | QR 코드 생성하여 카메라로 스캔 접속 |
| 맥락 없이 갑자기 난이도 오름 | Stage 1에서 요정 플루 튜토리얼로 메카닉 하나씩 소개 |

### 확정되지 않은 작은 결정들 (구현 단계에서 정리)

- 서버 포트 번호 기본값
- 스테이지 간 저장(같은 세션 내) 방식
- 효과음 수집/제작 계획
- README에 담길 교사용 실행 가이드 상세

## 11. 용어집

- **비트(Beat)**: 한 스테이지 안의 독립적인 퍼즐 구역 (체크포인트 단위)
- **권한 있는 서버(authoritative server)**: 서버가 게임 상태의 유일한 진실 원천인 방식
- **Client Isolation**: 같은 Wi-Fi에 연결된 기기끼리 통신을 차단하는 보안 설정
- **Tick**: 서버의 게임 루프 1회분 (MVP에서 30fps)
- **상태변화**: 물의 고체/액체/기체 간 상호 전환 (녹음·얾·끓음·증발·응결·승화)
